# 回滚策略

本文档定义了教学管理系统在部署、数据库变更、配置更新等场景下的回滚策略与操作流程。

---

## 1. 部署回滚

### 1.1 K8s 部署回滚

```yaml
# 回滚到上一版本
kubectl rollout undo deployment/edu-user

# 回滚到指定版本
kubectl rollout undo deployment/edu-user --to-revision=3

# 查看回滚状态
kubectl rollout status deployment/edu-user
```

### 1.2 Docker 镜像回滚

```bash
# 查看镜像历史
docker history edu-system/edu-user:latest

# 回滚到指定镜像
docker pull edu-system/edu-user:<previous-tag>
docker tag edu-system/edu-user:<previous-tag> edu-system/edu-user:latest
docker push edu-system/edu-user:latest
```

### 1.3 自动回滚触发条件

```yaml
# Harness 流水线配置
pipeline:
  stages:
    - stage:
        name: Deploy
        spec:
          execution:
            steps:
              - step:
                  type: K8sRollingDeploy
                  spec:
                    skipDryRun: false
                    # 失败自动回滚
                    rollback:
                      enabled: true
```

## 2. 数据库回滚

### 2.1 回滚脚本规范

每个服务的回滚脚本存放在 `harness/changes/{服务名}/rollback.sql`：

```sql
-- edu-user 回滚脚本
-- 版本: 1.0.0
-- 执行前请备份数据！

SET FOREIGN_KEY_CHECKS = 0;

-- 清空表数据（保留表结构）
TRUNCATE TABLE sys_user;

-- 或完全删除表
-- DROP TABLE IF EXISTS sys_user;

SET FOREIGN_KEY_CHECKS = 1;
```

### 2.2 数据恢复流程

```bash
# 1. 备份当前数据
mysqldump -u root -p edu_user > /backup/edu_user_$(date +%Y%m%d_%H%M%S).sql

# 2. 执行回滚脚本
mysql -u root -p edu_user < rollback.sql

# 3. 验证数据
SELECT * FROM sys_user LIMIT 10;
```

### 2.3 增量回滚

对于需要保留部分数据的场景：

```sql
-- 删除指定时间范围的数据
DELETE FROM sys_user 
WHERE create_time > '2024-01-01 00:00:00' 
AND create_time < '2024-01-02 00:00:00';

-- 回滚到指定状态
UPDATE course SET status = 'DRAFT' WHERE id = 1;
```

## 3. 配置回滚

### 3.1 Nacos 配置回滚

```bash
# 查看配置历史
curl -X GET "http://nacos:8848/nacos/v1/cs/history?dataId=edu-user.yml&group=DEFAULT_GROUP"

# 回滚到指定版本
curl -X POST "http://nacos:8848/nacos/v1/cs/history/rollback" \
  -d "dataId=edu-user.yml&group=DEFAULT_GROUP&nacostenant=&version=5"
```

### 3.2 本地配置回滚

```bash
# Git 回滚配置文件
git checkout HEAD~1 -- config/application.yml
```

### 3.3 环境变量回滚

```yaml
# 使用 Secret 或 ConfigMap 的历史版本
apiVersion: v1
kind: Secret
metadata:
  name: db-config
  namespace: education
data:
  # 使用上一版本的 base64 编码
  password: <previous-base64>
```

## 4. 事务回滚

### 4.1 编程式事务

```java
@Service
public class CourseServiceImpl implements ICourseService {

    @Transactional(rollbackFor = Exception.class)
    @Override
    public void saveCourse(Course course) {
        try {
            courseMapper.insert(course);
            // 更新教师课程数
            teacherService.incrementCourseCount(course.getTeacherId());
        } catch (Exception e) {
            // 手动回滚
            TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
            throw new ServiceException("保存课程失败");
        }
    }
}
```

### 4.2 分布式事务回滚 (Seata)

```java
@GlobalTransactional(timeoutMills = 30000, name = "save-course")
public void saveCourse(Course course) {
    // 分支事务
    courseMapper.insert(course);
    
    // 某步骤失败会自动回滚全局事务
    teacherService.incrementCourseCount(course.getTeacherId());
}
```

## 5. 回滚检查清单

### 5.1 回滚前检查

| 检查项 | 说明 |
|--------|------|
| 数据备份 | 确认已备份当前数据 |
| 影响范围 | 评估回滚影响范围 |
| 通知相关方 | 提前通知相关人员 |
| 回滚时间 | 安排在低峰期执行 |

### 5.2 回滚后验证

| 检查项 | 说明 |
|--------|------|
| 服务状态 | 确认服务正常运行 |
| 数据一致性 | 验证数据完整性 |
| 核心功能 | 测试核心业务流程 |
| 日志检查 | 查看异常日志 |

## 6. 紧急回滚流程

### 6.1 一键回滚脚本

```bash
#!/bin/bash
# rollback.sh - 紧急回滚脚本

SERVICE_NAME=$1
VERSION=$2

echo "开始回滚服务: $SERVICE_NAME 到版本: $VERSION"

# 1. 回滚 K8s 部署
kubectl rollout undo deployment/$SERVICE_NAME

# 2. 等待部署完成
kubectl rollout status deployment/$SERVICE_NAME

# 3. 执行数据库回滚
mysql -u root -p$MYSQL_PASSWORD edu_$SERVICE_NAME < ./harness/changes/$SERVICE_NAME/rollback.sql

# 4. 验证服务
curl -f http://$SERVICE_NAME/health || exit 1

echo "回滚完成"
```

### 6.2 回滚执行流程

```
发现异常
    ↓
评估影响范围
    ↓
启动回滚（决策者确认）
    ↓
执行回滚脚本
    ↓
验证服务状态
    ↓
通知相关方
    ↓
问题分析
```

## 7. 回滚策略对比

| 策略 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 蓝绿部署 | 回滚快速，无停机 | 需要双倍资源 | 生产环境 |
| 滚动更新 | 资源占用小 | 回滚较慢 | 日常部署 |
| 金丝雀发布 | 风险可控 | 复杂度高 | 新功能发布 |
| 灰度发布 | 精细控制 | 需要额外配置 | 大规模系统 |
